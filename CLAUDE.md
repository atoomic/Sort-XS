# CLAUDE.md — Sort::XS

## What is Sort::XS

A Perl XS module providing fast C-implemented sorting algorithms for one-dimensional arrays of integers or strings. Algorithms: QuickSort (default), HeapSort, MergeSort, ShellSort, InsertionSort.

## Build and Test

```bash
perl Makefile.PL && make && make test
```

For XS modules, `make` (compilation) and `make test` are separate steps — `make test` alone won't recompile C changes.

To run a single test:
```bash
prove -v t/01_sort.t          # won't work for XS
perl -Iblib/lib -Iblib/arch t/01_sort.t   # correct way
make test                      # or just run all tests
```

Benchmarks require `DEVEL_TESTS=1` and optional modules (Sort::Key, Sort::Key::Radix, Benchmark).

## Architecture

```
lib/Sort/XS.pm   — Perl API: xsort(), ixsort(), sxsort() wrappers
XS.xs            — XS bindings: _jump_to_sort() + per-algorithm XS functions
sort.c           — C implementations of all sort algorithms
sort.h           — Type definitions (ElementType union, CmpFunction) and prototypes
XS.c             — Auto-generated by ExtUtils::ParseXS from XS.xs (do NOT edit)
```

### Data flow

1. Perl calls `xsort([1,5,3])` which routes to the appropriate XS function
2. `_jump_to_sort()` in XS.xs: unmarshals Perl AV to C `ElementType[]` array
3. C sort function sorts the array in-place
4. `_jump_to_sort()` marshals results back into a new Perl AV

### Key types

```c
typedef union {
    double f;   // unused on master, reserved for float sorting
    IV     i;   // integers (Perl IV)
    char  *s;   // strings (pointer into SV buffer)
} ElementType;

typedef int (CmpFunction)(const ElementType *a, const ElementType *b);
```

## Adding a new sort algorithm

1. Implement `void YourSort(ElementType A[], int N, CmpFunction *cmp)` in `sort.c`
2. Declare it in `sort.h`
3. Add to the `SortAlgo` enum in `XS.xs` (order must match `sort_function_map[]`)
4. Add to `sort_function_map[]` in `XS.xs`
5. Add XS function declarations (integer + string variants)
6. Add to `$_mapping` in `lib/Sort/XS.pm`
7. Add tests in `t/01_sort.t`

## Adding a new data type

1. Add a comparison function in `sort.c` (e.g., `compare_float`)
2. Declare it in `sort.h`
3. Add to `SortType` enum and `cmp_functionmap[]` in `XS.xs`
4. Add marshalling in `_jump_to_sort()`: input loop (SvNV) and output loop (newSVnv)
5. Add XS function declarations with the new suffix (e.g., `_float`)
6. Wire into `$_mapping` in `XS.pm`

## Conventions

### Memory management
- Use `Newx(ptr, count, Type)` / `Safefree(ptr)` — never raw `malloc`/`free`
- Newx integrates with Perl's memory debugging and croaks on OOM
- **Never use VLA** (variable-length arrays) on stack — use heap allocation with Newx

### croak() safety
- `croak()` does a `longjmp` — any SV allocated before croak with refcount>0 leaks
- Validate inputs BEFORE allocating output SVs
- If you must allocate first, use `sv_2mortal()` to ensure cleanup

### Integer arithmetic
- Midpoint calculation: use `Left + (Right - Left) / 2` to avoid overflow (not `(Left + Right) / 2`)
- Array sizes come from `av_len()` which returns `SSize_t` — be aware of truncation when storing in `int`

### Comparisons
- Prefer branchless integer comparison: `(a > b) - (a < b)`
- String comparison uses `strcmp()` directly

### Performance patterns
- Pre-extend output AV with `av_extend(av, size)` before `av_push` loop
- Use `AvARRAY(input)` for direct SV** access instead of per-element `av_fetch()`
- Hoist type checks outside loops (separate int/str loops, not per-element branching)
